Harness Engineering 与主流 AI Coding 工作流横向对比

本页比较 Harness Engineering 标准插件、Spec Workflow MCP、GitHub Spec Kit、BMAD Method、Agent OS、Kiro Specs 和 Task Master。重点不是谁“更强”，而是它们分别解决哪一层问题，以及你的 Harness Engineering 应该如何定位、组合和演进。

一句话结论 Harness Engineering 更像企业交付治理层，可以吸收其他 spec 或 task 工具的产出，再统一做风险、评审、验证和审计。
最强差异点 10 阶段审计链、riskTags、gateProfile、extensionStages、validator、执行与评审分离。
当前短板 生态、可视化、社区成熟度和一键安装体验还弱于 BMAD、Spec Kit、Spec Workflow MCP 这类成熟项目。

Harness Engineering 标准插件

企业级 AI Coding 交付治理层，把需求、实现、评审、测试、CI、部署验证和最终审计串成可验证链路。

本地插件 中文命令 风险自适应 审计证据

典型流程

流程配置 → 变更初始化 → 需求分析 → 需求评审 → 设计准备 → 实现 → 代码评审 → 测试与 CI → 部署验证 → 最终审计

优点

  • 覆盖完整交付链，不只覆盖 spec 阶段。
  • riskTags、gateProfile、extensionStages 能区分轻量和加严流程。
  • 强制执行 Agent 与评审 Agent 或评审流程分离。
  • 证据落在 `.harness/changes/<change-id>/`，适合审计追溯。
  • validator 能把部分质量门禁变成可执行检查。

短板

  • 生态和社区还在第一版阶段。
  • 缺少 dashboard、任务状态 UI、IDE 原生体验。
  • 安装脚本和中文资产还需要打磨编码、版本升级和冲突合并体验。
  • 流程偏严，新人需要好的 README、skill 和命令封装降低心智负担。

适用场景

政企交付、金融金额计算、权限安全、数据库 migration、生产部署、外部验收、多团队协作、需要交付审计的 AI Coding 场景。

慎用场景

极小个人脚本、一次性 demo、无需留痕的探索性编码。此时可只用 fast-iteration，但仍保留最低审计责任。

Spec Workflow MCP

MCP server 形态的 spec-driven 工作流，强调需求、设计、任务、实现的顺序推进、审批和可视化进度。

MCP Dashboard Approval 多工具

典型流程

Requirements → Design → Tasks → Implementation，配合 dashboard 做进度与审批管理。

优点

  • 作为 MCP server，接入 Claude、Cursor、Codex 等工具较自然。
  • dashboard 和审批系统对团队可见性很友好。
  • 比纯文档模板更容易让 AI 按阶段推进。
  • 适合将 spec 工作流产品化给个人或小团队使用。

短板

  • 重心在 spec 和任务流，不是企业全链路交付审计。
  • 高风险规则、部署验收、失败沉淀等治理能力需要二次扩展。
  • 依赖 MCP 服务和 dashboard，运维形态比纯文件资产更重。
  • GPL-3.0 许可对部分闭源商业集成需要谨慎评估。

适用场景

希望快速获得 spec 工作流、任务追踪、审批和可视化面板的团队，尤其是 MCP 生态用户。

慎用场景

对合规审计、部署验证、风险分级、评审分离有硬要求的企业交付场景，需要额外治理层。

GitHub Spec Kit

开源的 Spec-Driven Development toolkit，强调先定义要构建什么，再生成 plan、tasks 和实现。

SDD Specify CLI 多 Agent 集成 MIT

典型流程

Constitution → Specify → Plan → Tasks → Implement，可通过 extensions 和 presets 扩展。

优点

  • 方法论清晰，是当前 SDD 代表性工具之一。
  • 支持大量 AI coding agent 集成，社区扩展和 presets 体系活跃。
  • constitution、spec、plan、tasks 的文档链路适合规范驱动开发。
  • MIT 许可对企业采用友好。

短板

  • 主要解决 spec 到 implement，不天然覆盖部署验证、UAT 和最终交付审计。
  • 强依赖用户和 Agent 正确理解 SDD，复杂项目可能产生较大文档和 token 成本。
  • 风险标签、按风险收紧门禁、执行评审分离需要扩展实现。

适用场景

绿地项目、规范先行团队、多工具 AI coding 统一、希望把“需求先于代码”制度化的研发组织。

慎用场景

只要极快改一个小 bug，或强监管交付需要现成审计链时，单独使用 Spec Kit 还不够。

BMAD Method

AI-driven agile development framework，使用专门 Agent、工作流和规划轨道，从分析、规划、架构到故事级实现。

多 Agent Agile PRD Architecture

典型流程

Analysis → Planning → Solutioning → Implementation，按 Quick / Method / Enterprise 轨道调整深度。

优点

  • 角色模型丰富，PM、Architect、Dev、UX、QA 等分工自然。
  • 对产品规划、PRD、架构、story 级开发支持强。
  • scale-adaptive 思路成熟，能从小修到企业平台切换深度。
  • 社区、文档、模块化生态较完整。

短板

  • 流程和 Agent 多，新人容易先学系统而不是先交付需求。
  • 更像 AI agile 方法框架，不是交付审计 validator。
  • 企业合规证据、部署验收和风险规则需要定制模块补齐。

适用场景

从 0 到 1 产品、复杂功能规划、跨角色协同、需要产品经理和架构师 Agent 深度参与的项目。

慎用场景

已有明确需求，只需要轻量审计和验证闭环时，BMAD 可能显得偏大。

Agent OS

轻量系统，用于发现、组织并注入团队标准，帮助 AI 工具按项目习惯生成更好的 spec 和计划。

Standards Spec Shaping Profiles 轻量

典型流程

Plan Product → Discover Standards → Inject Standards → Shape Spec → 交给 IDE Plan Mode 实现。

优点

  • 专注“团队如何写代码”的标准注入，边界清晰。
  • 全局 profile 与项目安装分离，适合多项目复用。
  • discover-standards 能把隐性约定变成显性文档。
  • 相对轻量，不强行接管整个开发流程。

短板

  • v3 更聚焦标准和 spec shaping，弱化实现编排。
  • 不提供完整 CI、部署、验收审计链。
  • 对需要硬门禁和证据目录的企业交付需要叠加其他机制。

适用场景

已有开发流程，但 AI 经常不遵守项目约定；希望把编码规范、架构偏好和项目知识注入 Agent。

慎用场景

团队还没有稳定标准，或需要交付审计而不只是上下文注入时，单独使用 Agent OS 不够。

Kiro Specs

IDE 原生 specs、steering 和 hooks，面向从需求、设计、任务到执行的结构化开发体验。

IDE 原生 Specs Steering Hooks

典型流程

Requirements / Bug Analysis → Design → Tasks → Task Execution，可结合 steering 与 hooks。

优点

  • IDE 内建体验好，新人不必拼装多个命令行工具。
  • requirements、design、tasks 三件套清晰。
  • 支持 task execution 状态、并行任务分析、steering 常驻上下文。
  • hooks 可自动化部分重复动作。

短板

  • 绑定 Kiro IDE，跨工具复用不如纯文件插件灵活。
  • 企业自己的审计格式、风险规则和最终交付证据仍需外部约束。
  • 对已有 IDE 标准化团队，迁移成本需要评估。

适用场景

团队愿意统一到 Kiro IDE，希望 specs、任务执行、steering 和 hooks 都有一体化体验。

慎用场景

多 IDE、多 Agent 混合团队，或要求流程资产完全工具无关的企业交付。

Task Master

AI task management 工具，把 PRD 解析成任务、依赖、优先级、测试策略，并支持复杂度分析和 MCP 调用。

PRD → Tasks MCP 依赖管理 复杂度分析

典型流程

写 PRD → parse-prd → tasks.json → analyze-complexity → expand → next_task → set_task_status。

优点

  • 任务拆解、依赖、状态跟踪能力强。
  • 适合把大需求拆成可执行小步，避免 Agent 漫游。
  • MCP 工具覆盖任务读取、更新、扩展、复杂度分析、依赖校验。
  • 可以作为 Harness 或 Spec Kit 后面的任务执行管理层。

短板

  • 核心是任务管理，不是需求评审、代码评审或交付审计框架。
  • PRD 质量决定任务质量，前置需求澄清仍要靠团队。
  • 验证证据、部署回滚、用户确认需要外部流程补齐。

适用场景

已有 PRD，想快速拆任务、排依赖、看复杂度、驱动 Agent 按下一任务推进的团队。

慎用场景

需求本身还模糊，或要求强审计、强评审、强验证闭环时，不能只靠任务拆解。

关键维度速览

维度 Harness Engineering 市场方案普遍表现 建议
需求到交付链路 覆盖 10 阶段审计链，并含部署验证、用户确认、最终审计。 Spec Kit、Spec Workflow、Kiro 更集中在 spec、design、tasks、implementation。 保留全链路定位，不要退化成另一个 spec 生成工具。
风险自适应 内置 `riskTags`、`gateProfile`、`riskRules` 和最小验证命令计算。 BMAD 有 scale-adaptive，Spec Kit 有 extensions，但风险标签到强制门禁通常需要定制。 把风险配置做成 Harness 的招牌能力，继续增强可视化和自动校验。
执行与评审分离 流程明确要求实现 Agent 与评审 Agent 或评审流程分离。 BMAD 有多角色，部分工具有 review 扩展，但不一定形成交付硬约束。 继续强化“谁实现、谁评审、证据在哪里”的追踪。
新人上手 中文命令友好，但插件和命令体验仍需打磨。 Kiro IDE 原生体验最好，Spec Workflow MCP dashboard 可视化好，BMAD 文档丰富。 优先做 `harness install`、`harness doctor`、`harness next` 和中文交互向导。
验证闭环 有 validator、CI 证据、deploy-result、acceptance audit。 多数方案会生成任务或测试建议，但未必强制落地执行证据。 继续把“没有验证证据不得交付”做成机器可检查规则。
生态成熟度 第一版本地插件，生态刚起步。 Spec Kit、BMAD、Spec Workflow MCP 已有较高社区关注和工具集成。 先把 Harness 定义为可叠加治理层，兼容这些工具的产物。

组合使用建议

Harness + Spec Kit

用 Spec Kit 生成 spec、plan、tasks；用 Harness 做风险分级、评审分离、CI、部署验证和最终交付审计。适合规范驱动和企业交付兼顾的团队。

Harness + BMAD

用 BMAD 做产品、角色、架构和 story 规划；用 Harness 接住每个 story 的证据、验证和验收。适合从 0 到 1 的复杂平台项目。

Harness + Task Master

用 Task Master 管任务、依赖、复杂度和下一步；用 Harness 管门禁、评审、验证和交付审计。适合已经有 PRD 的持续交付团队。

对 Harness Engineering 的产品化建议

优先级 建议 原因
P0 把安装体验收敛成 `harness install` 或单行 `npx`。 当前对新人仍像“复制 starter 资产”，比成熟工具安装心智更高。
P0 新增 `harness doctor`、`harness next <change-id>`、`harness audit <change-id>`。 新人最常问的是“现在该做什么”和“能不能交付”。
P1 把风险配置结果输出为结构化 JSON 和 Markdown 双格式。 方便 validator、dashboard、CI 和其他工具读取。
P1 支持导入 Spec Kit、BMAD、Task Master 的产物。 把 Harness 定位成上层治理层，而不是和成熟 spec 工具硬拼。
P2 做一个只读 dashboard 展示 change 证据完整度。 Spec Workflow MCP 和 Kiro 的可视化优势明显，Harness 可先从审计可视化切入。
